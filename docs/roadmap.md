# 开发路线图

## 游戏目标与基调

囚烬日记是一款**叙事驱动的 Roguelike 回合制 RPG**。核心体验由两个维度交织：

**Roguelike 跑程**：每局随机地图，多类型节点分支探索，节点产出技能、遗物、货币；死亡重置局内一切，带着记忆印记从原点重来。成长来自组合与交互，而非数值堆叠——技能选择、流派搭配、遗物协同构成每局不同的构建。

**元进度叙事**：跨局积累决定故事深度。每一次探索拼凑他人留下的记忆碎片，也在拼凑自己。图鉴积满时，最后的拼图指向余烬——主角的另一块碎片。终局选择决定五条不同的结局，其重量来自多局积累的关系与真相。

**整体基调**：克制。世界的规则不需要解释，真相不会被说破，叙事通过细节渗透。SAN（理智）是唯一会在战斗中被磨损的东西——越低越接近渊虫，归零意味着失去自我。玩家永远有得选，任何系统不允许强制等待或强制结果；运气也是实力的一部分，选择做出后不可回头。

## 架构原则

以下原则来自引擎与游戏层的设计决策，指导后续所有开发：

**分层方向单向**：`UI → Store → Game → rpg → engine`，下层不感知上层；跨层依赖只允许向下，禁止反向。`@xwink/engine / @xwink/rpg` 是主体，`apps/game` 是验证其能力的业务 demo。

**插件体系**：新业务系统以 `EnginePlugin` 接入——声明依赖，引擎拓扑排序后初始化；SAN 系统、未来的 AI 系统均走这条路，不改引擎核心。

**跨层只传数据，不传实例**：层间通信只传 `Data` / `Snapshot` 类型对象；`UnitContainer` 是唯一的实例组装点，其他层不持有 `Unit` 实例。

**二层事件系统**：`engine.on/emit` 同步执行，用于战斗逻辑（buff 挂载、伤害计算）；`engine.queue/consume` 缓冲，用于日志和 UI 更新；两套系统不混用。

**BattleMode 热拔插**：战斗决策是可替换接口；`AutoBattleMode`（主线）和未来的 `ManualBattleMode`（扩展玩法）通过同一接口接入，引擎不感知具体策略。

**Sandbox 快照**：战斗状态支持 `commit / undo / redo`；存档恢复通过重建引擎实例后 hydrate 各子容器实现，相同 seed 保证相同推演结果。

**确定性 RNG**：随机性按层分桶——战斗内 RNG 由 BattlePlugin 自治（战斗 seed），session 编排层（节点变异 / 事件抽取 / 商店洗牌 / 遗物三选一）另起一条 Mulberry32 序列（session seed）。两条序列均可序列化：战斗 seed 写入 sandbox 快照；session seed 与已消耗 count 写入存档，恢复时空跑 count 步把 PRNG 推回原位置。seed 固定则整局推演确定，支持录像回放、存档重开复现、调试复现。

**Token 渲染边界**：逻辑层只暴露结构化 Token，不产出自然语言字符串；渲染层负责 Token → 文本转换，不同语言或渲染引擎只替换这一层；Token 语言无关，是 i18n 的天然边界。`@xwink/rpg` 的 `describe` 模块已遵循此结构，其他描述文本对齐同一规范。

**i18n 分层**：UI 层通过 `i18n/useT()` 组合式函数获取当前 locale 的字符串，locale 状态由 `store/settings.ts` 统一持有；Token 渲染边界负责战斗描述的语言无关性；两层合力覆盖全部文本，不在组件内硬编码多语言分支。

## 设计约定

### 美术风格

整体美术风格参照**饥荒（Don't Starve）**，核心特征如下：

| 维度     | 描述                                                                  |
| -------- | --------------------------------------------------------------------- |
| 线条     | 厚重手绘轮廓线（描边宽 3–6px），棱角分明，无抗锯齿滑边                |
| 色调     | 以黑褐色为主体，局部用深红/橙黄做冷暖点缀；整体低饱和度，避免鲜艳色块 |
| 质感     | 类炭笔 / 油墨质感，允许纹理颗粒；拒绝渐变光泽感                       |
| 比例     | 轻度夸张变形——角色头部略大，四肢偏瘦长，配合场景压迫感                |
| 背景     | 剪影式构图，远景高度简化，留黑暗空间给玩家想象                        |
| 装饰密度 | 克制；每帧只有 1–2 个视觉焦点，不堆砌细节                             |

> 所有美术资源（角色立绘、场景背景、图标）均应遵循此风格规范。
> 资源详细尺寸与 AI 生成提示词见 `assets.md`。

### 节点路线图

- **分支路径**：玩家在节点间主动选择路线，同一层内存在多条可行路径
- **终点固定**：每层终点为 Boss 节点，结构不随机
- **战斗节点敌人类型**：普通战斗节点随机抽取普通敌人；精英节点抽取精英敌人
- **怪物变异词条**：敌人生成时按概率随机附加变异词条，词条影响两侧——能力增幅（战斗属性/行为强化）和掉落增幅（遗物品质/数量提升）；词条本质是挂载在单位上的 buff，由 rpg 层支持，词条定义和触发在 game 层管理

### 技能流派

四大流派各有核心哲学与子流派分支，克制关系循环：

| 流派     | 哲学                         | 子流派                                                       |
| -------- | ---------------------------- | ------------------------------------------------------------ |
| **均衡** | 不把鸡蛋放在同一个篮子里     | 控制流（减益+DOT）/ 自由流（混搭协同加成）                   |
| **猛攻** | 等待就是把主动权交出去       | 连击流（高速多段+额外行动）/ 蛮力流（极限数值碾压）          |
| **龟缩** | 让对手先犯错，然后一次收回来 | 反伤流（受击反弹）/ 护盾流（叠加吸收）                       |
| **赌狗** | 运气也是实力，不赌不知道     | 概率流（依赖概率触发爆发）/ 极限流（扩大倍率区间换极高上限） |

克制关系：猛攻 → 赌狗 → 龟缩 → 猛攻（循环）；均衡无克制无被克，是混搭核心。

skill delta 是子流派分支的唯一入口，基础技能本身不区分子流派；极限流期望值不得高于概率流上限；赌狗概率技能需明确显示期望值，避免纯随机感。

### 货币与成就

- **通用货币**：战斗掉落为主要来源；流向商店、强化节点、技能抽奖等消耗点
- **稀有货币**：成就系统专属产出；流向全局解锁（称号、特殊元进度内容等）；不参与日常消耗循环
- **成就系统**：渊对特定行为的应答——做过了，它记住了；不是奖励，是记录。触发条件达成后给予稀有货币 + 称号；称号在叙事层有语义，不只是装饰

### 存档

- **槽位数量**：与结局条数一致（五条结局对应五个存档槽），不扩容；每槽可独立命名
- **操作集合**：保存、读取、覆盖、删除、导入、导出——全部可用；导入导出保证跨设备与跨版本数据迁移后游戏进度一致
- **格式版本化**：存档结构携带版本号；升级时就地迁移至最新格式，迁移失败回退而非静默损坏
- **逻辑隔离**：存档层是纯 I/O 边界，逻辑层不感知存储媒介（localStorage / 文件 / 后端 API 均可替换）；存档内容是状态快照，不包含可推导的中间计算结果

### 本地化

- **语言范围**：中文（默认）+ 英文
- **i18n 接口**：`i18n/useT()` 返回响应式翻译函数，locale 切换时组件自动重渲染；locale 持久化在 `store/settings.ts`（`crucible:settings`）
- **词条管理**：`i18n/zh.ts` 导出 `TKey` 作为全局 key 集合；`i18n/en.ts` 类型强制为 `Record<TKey, string>`，编译期确保不漏译
- **迁移策略**：设置界面已双语；存量界面文本在 Phase 2 随功能改动逐步迁移至 `useT()`，不做一次性全量替换

### 设置

- **全局设置**：语言（zh/en）、BGM 音量（0–100）、音效音量（0–100）、减少动效（关闭非必要动画/过渡）
- **战斗设置**：战斗速度（慢 800ms / 标准 400ms / 快 150ms，驱动时间轴动画时长）、战斗日志显示
- **持久化**：独立 key `crucible:settings`，与游戏存档解耦；容错加载，字段缺失时静默回退默认值
- **减少动效**：通过 `html.reduced-motion` class 压制全局 `animation-duration` 和 `transition-duration`，无需组件逐一适配

## 四个主线阶段

### Phase 1 · 能玩 ✓

**目标**：一局完整的荒渊跑程从头到尾可以玩通。

**完成标志**：

- [x] 玩家可命名角色
- [x] 预设固定地图上可推进（战斗 / 事件 / 商店 / 休息节点）
- [x] 荒渊 Boss 节点可触发并结算
- [x] HP 归零与 SAN 归零是两条独立失败路径
- [x] SAN 变化影响战斗属性（伤害增幅 / 减伤损失随 SAN 曲线变化）
- [x] 对应结局画面可展示
- [x] 遗物可在战斗后选取（当局加成生效）
- [x] 技能拾取节点可选取并生效
- [x] 通用货币可从战斗获取并在商店消耗
- [x] 怪物变异词条基础实现（2-3 条）
- [x] 刷新后存档不丢（localStorage v4：phase / 节点 / 战斗 sandbox / sessionRng 全部恢复，旧版本号自动清理，story 与 result 不复原）
- [x] 设置界面（全局：语言 / BGM / 音效 / 减少动效；战斗：速度 / 日志显示）+ 独立持久化

- [ ] 成就系统骨架可触发，稀有货币奖励可产出

- 元进度只解锁可能性，不缩短通关难度

**完成标志**：

- [ ] 余烬在枢纽的对话随关卡进度递进（七幕完整）
- [ ] NPC 记忆碎片按设计数量实装并可收集
- [ ] 驻渊四 NPC 好感度积累机制可用，碎片序列可解锁
- [ ] 遗物首次获取后永久收录图鉴
- [ ] 成就称号完整
- [ ] 图鉴 100% 时触发闪回和隐藏槽位
- [ ] storyline.md 的触发节点全部接入对应系统钩子
- [ ] 天渊·先遣进入终局前台词阶段

**方向**：

- 叙事不解释，只留线索——任何文本不说破真相
- 触发节点是系统事件，不是写死的弹窗
- 余烬弧光的反转拐点必须在壁渊段感受到，不能晚于驻渊

### Phase 4 · 有结局

**目标**：五条终路全部可触发；游戏完整可发布。

**完成标志**：

- [ ] 终局战斗可触发（含隐藏被动开关）
- [ ] 永久封号路径真实可达
- [ ] 相认后游戏状态永久变化
- [ ] 所有结局分支通过测试
- [ ] 内容与数值收尾完毕
- [ ] 数值平衡专项（技能 / 遗物 / SAN 曲线，结合外部测试数据迭代）
- [ ] 外部玩家测试（至少一轮；测试者不了解设计背景，能玩完并理解故事）
- [ ] 字体与音频资源授权确认（所有商业用途许可到位）
- [ ] 内容分级（IARC 或适用地区评级完成）
- [ ] Steam 商店页完备（Capsule 图三规格、截图 ≥5 张、宣传片、标签）
- [ ] Steam 成就接入（Steamworks SDK，与游戏内成就系统对齐）
- [ ] 崩溃上报接入（Sentry 或同等方案）

**方向**：

- 结局差异靠叙事密度，不靠新系统
- 永久封号没有后门
- 发布标准：不了解设计背景的人能玩完一局并理解故事在讲什么

## 扩展阶段（主线后）

### Phase 5 · 扩展层

以下方向互相独立，可并行或按需推进：

- [ ] **手动战斗**：ManualBattleMode 在 rpg 层实现，game 层选择模式；主线叙事体验不受影响，走独立入口
- [ ] **PixiJS 渲染**：以战斗场景为切入点，逐步替换 Vue / DOM
- [ ] **持久化服务**：引入独立存档服务层（结构化本地或后端 API）；替换 localStorage 调用点；存档格式在主线阶段已保持可迁移
- [ ] **Tauri 封装 + Steam 上架**：用 Tauri 打包为桌面可执行文件；接入 Steamworks SDK 实现成就、云存档等平台功能
- [ ] **美术资源**：在此阶段开始考虑；Phase 1 待产出清单见 `assets.md`

## 阶段顺序的依据

没有可玩的跑程，一切验证都是空谈，所以 Phase 1 先于一切。

世界层级是叙事触发的容器，元进度是故事深度的骨架，所以 Phase 2 先于 Phase 3。

终局选择的重量来自前三阶段的积累，所以 Phase 3 先于 Phase 4。

发行准备（授权、评级、外部测试、平台接入）与内容收尾并行推进，同属 Phase 4 的完成条件，不单独成阶段。

手动战斗不影响主线叙事，持久化和渲染层替换不影响业务逻辑，PixiJS 和 Tauri 是内容完成后的技术升级与发行准备——四者均归入主线完成后的扩展，所以 Phase 5 独立于主线之外。
